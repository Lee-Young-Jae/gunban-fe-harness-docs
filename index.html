Gunban 하네스 구조v2.1

코드 작업 요청을 4종 파이프라인으로 분류·실행하는 멀티 에이전트 하네스
에이전트 8 스킬 5 파이프라인 4 스크립트 9 최종 갱신 2026-05-14

01개요

Gunban 하네스는 프론트엔드 개발 요청의 단일 진입점 dev-flow 스킬과, 그 아래 의도를 4종 파이프라인으로 분류·실행하는 멀티 에이전트 시스템이다. 각 작업은 슬러그 기반 워크스페이스(_workspace/{date}_{slug}/)에 격리되어 동시 진행이 가능하며, 모든 산출물은 작업 종료 시 work-history/로 영구 보존된다.

핵심 설계 원칙

1. 단일 진입점

모든 코드 작업 요청은 dev-flow를 통과한다. 직접 에이전트 호출 금지.

2. 슬러그 워크스페이스

동시 진행 가능. 각 작업이 자기 디렉토리에 격리된 산출물을 가진다.

3. Task 툴 기반 호출[1]

claude -p 서브프로세스 미사용. 캐시 절약 + 컨텍스트 격리.

4. 자동 컨텍스트 주입

retrieve-context.py가 prepare에서 관련 이력을 자동 검색.

5. 시각 검증 자동화

Playwright + Figma export + pixelmatch로 정량 diff. LLM의 픽셀 추측 X.

6. 승인 게이트 + 예산 가드

비싼 phase 후 체크포인트. 토큰 누적 80%/100%에서 사용자 의사결정.

에러 핸들링

06에이전트 카탈로그

intent-classifier haiku

파이프라인 라우팅. trigger 신호 점수 계산.

in: 사용자 메시지
out: JSON (pipeline + confidence + 후보)

planner opus

구현 계획 수립. FSD 레이어, API 매핑, 라우트 추정.

in: 00_parsed_input + 00_related_context
out: 00_plan.md

figma-analyzer opus

Figma 데이터 → 구현 스펙. 노드 위계 트리 + 정량 값 표 + 컴포넌트 매핑.

in: Figma URL (via MCP)
out: 01_figma_spec.md

page-builder opus

FSD 기반 코드 작성. 모든 파이프라인의 구현/수정 담당.

in: 00_plan + 01_figma_spec (or 보고서)
out: src/app/{route}/**

qa-inspector opus

3-way diff + visual-diff + 빌드 검증. 모든 파이프라인의 검증 담당.

in: 코드 + 01_figma_spec
out: 02_figma_diff.md + 03_qa_report.md

bug-locator opus

버그 위치 + 원인 가설 + 수정 범위. patch 파이프라인 첫 단계.

in: 00_user_intent (버그 보고)
out: 00_bug_report.md

diff-analyzer opus

신규 spec ↔ 기존 코드 6분류 차이. redesign 파이프라인 핵심.

in: 01_figma_spec + 기존 src
out: 01_redesign_diff.md

feedback-parser opus

피드백 항목화 + 직전 작업 매핑. refine 파이프라인 첫 단계.

in: 00_user_intent + 직전 task
out: 00_feedback.md
왜 intent-classifier만 haiku인가
단순 분류 작업. 깊은 추론 불필요. 응답 속도와 비용을 우선. 다른 모든 에이전트는 도메인 추론이 필요해 opus.

일상 사용 흐름

  1. 사용자가 자연어로 요청 ("메인 검색 페이지 만들어줘 + Figma URL")
  2. Claude가 dev-flow 스킬을 자동 트리거
  3. intent-classifier가 파이프라인 결정 → 사용자 한 줄 확인
  4. phase 순차 실행 + 체크포인트마다 사용자 승인
  5. qa 통과 후 archive 여부 선택 (archive / 유지 / 삭제)
  6. 최종 보고서로 산출물 + 토큰/시간 확인

흔한 실패 → 대응

상황대응
route 추정 신뢰도 낮음체크포인트 1에서 사용자 직접 입력
backpressure 실패1회 자기 수정 → 그래도 실패 시 fix_loop 진입
fix_loop max_iter 초과사용자 보고 + 종료. 남은 FAIL 직접 처리
diff-analyzer가 빈 라우트 발견redesign 중단 + greenfield 재시작 안내
refine 직전 슬러그 모호feedback-parser가 후보 제시 → 사용자 선택
visual-diff dev 서버 미가동자동 생략 + 보고서 표기
